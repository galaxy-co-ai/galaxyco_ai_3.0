10 Questions Engineers Will Ask

1. "Why Next.js over a separate frontend/backend?"

  • - Server Components reduce client bundle
  • - Streaming for Neptune responses
  • - API routes colocated with pages
  • - One deploy, one repo, one team

2. "How do you handle multi-tenancy?"

  • - Every table has workspace_id
  • - Drizzle query builders enforce scoping
  • - Application-level, not DB row-level security
  • - Middleware extracts tenant from auth

3. "Why Drizzle instead of Prisma?"

  • - Type inference without codegen
  • - SQL-like syntax, not abstracted away
  • - Better edge runtime support
  • - Lighter weight, faster cold starts

4. "How do you add new AI capabilities?"

  • - Single file: Zod schema + execute function
  • - Export from domain index
  • - Neptune auto-discovers tools
  • - No core changes required

5. "What happens when OpenAI is down?"

  • - Multi-model: OpenAI, Anthropic, Google
  • - Model router can failover
  • - Each model configured for specific tasks
  • - Response caching reduces dependency

6. "How do you handle real-time updates?"

  • - Pusher for WebSocket connections
  • - Channel per workspace for isolation
  • - SWR revalidation on events
  • - Optimistic UI where appropriate

7. "What's your testing strategy?"

  • - Vitest for unit tests
  • - Playwright for E2E
  • - TypeScript strict mode catches type bugs
  • - 63% coverage, targeting 80%+

8. "What technical debt exists?"

  • - 4 caching implementations (consolidating Q1)
  • - Multiple context builders (unifying Q1)
  • - API could be more RESTful (Q2)
  • - We track and plan for debt openly

9. "How does the AI tool system work?"

  • - 37+ tools with Zod-validated schemas
  • - Neptune parses intent, selects tools
  • - Tools execute with workspace context
  • - Results aggregated, streamed back

10. "How would I onboard to this codebase?"

  • - ARCHITECTURE.md explains system design
  • - TypeScript types are self-documenting
  • - Clear module boundaries (CRM, Finance, etc.)
  • - Start with one tool file to learn patterns

Architecture Overview

Key decisions and system design - reference during technical questions

Core Design Principles

1. Multi-tenant from Day One

Every table has workspace_id. Every query is scoped. No exceptions. Retrofitting multi-tenancy is expensive - we paid the cost upfront.

2. AI as Orchestration Layer

Neptune isn't a chatbot bolted on. The platform is designed around AI orchestration. Modules expose tools, Neptune chains them.

3. Type Safety as Documentation

TypeScript strict mode, zero errors. Types are the source of truth. Self-documenting code with compile-time bug prevention.

4. Progressive Disclosure

Features surface as users need them. New users see simplicity. Power users see depth. Complex software doesn't have to feel complex.

Key Technical Decisions

Drizzle ORM over Prisma

Type inference from schema without code generation. SQL-like syntax. Lighter weight. Better edge runtime support.

Trade-off: Smaller ecosystem

Multi-model AI (OpenAI + Anthropic + Google)

Provider redundancy. Model selection based on task. Cost optimization. Future flexibility.

Trade-off: More complex prompts

Tool-based AI over Prompt-only

Reliable execution (tools have schemas). Composable. Auditable. Adding a capability is a single file.

Trade-off: More upfront work

Serverless Postgres (Neon)

Scale to zero for cost efficiency. Branching for previews. Standard Postgres compatibility. Autoscaling without ops.

Trade-off: Cold starts

When Engineers Ask "Why?"

Q:

"Why Drizzle over Prisma?"

"Type inference without code generation, SQL-like syntax our team knows, and better edge runtime support. Trade-off is smaller ecosystem, but we accepted that."

Q:

"How do you handle multi-tenancy?"

"Every table has workspace_id. Every query is scoped through Drizzle's query builder. It's not row-level security in Postgres - it's application-level, but enforced consistently."

Q:

"How do you add new AI capabilities?"

"Single file with a Zod schema and execute function. Export it from the domain index. Neptune picks it up automatically. No core changes needed."

Q:

"What technical debt exists?"

"We have 4 caching implementations that should be one. We know it, it's documented, and consolidation is planned for Q1. We track debt openly."
